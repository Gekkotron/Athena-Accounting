# Frontend i18n Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the Athena-Accounting React frontend fully translatable into English and French (extensible to more languages), with English as the source of truth and French as the initial second locale. Each page migrates independently; the app stays 100% functional in French throughout.

**Architecture:** `i18next` + `react-i18next` + `i18next-browser-languagedetector`. Per-page namespace JSON files under `frontend/src/locales/{en,fr}/<namespace>.json`. Semantic keys (not English-worded). Persisted via `localStorage.athena.lang` with `navigator.language` fallback. No backend changes.

**Tech Stack:** React 18 + Vite + TypeScript, i18next v23, Vitest for tests. Existing Tailwind theming and font stack unchanged.

## Migration recipe (Tasks 2–11 share this shape)

Each Phase C task migrates one namespace. The procedure is the same; only the files and namespace name differ.

1. **Grep the feature's directory for French string literals.** In the target directory (e.g. `frontend/src/pages/Dashboard/`), find:
   - JSX text nodes: `>Texte FR<`
   - JSX attributes: `title="…"`, `placeholder="…"`, `aria-label="…"`, `alt="…"`
   - Toast / alert / confirm text: `toast.success('…')`, `alert('…')`, `confirm('…')`, `throw new Error('…')` (only user-visible ones)
   - String literals assigned to variables and rendered later
   - Skip: log messages, dev-only text, code identifiers, constants like `'FR'` that are locale codes not user-visible.

   Command patterns to seed the grep:
   ```
   grep -rn --include='*.tsx' --include='*.ts' -oE '>[A-ZÀ-ÿ][A-Za-zÀ-ÿ0-9 ,.:!?()/'\''-]{2,}<' <dir>
   grep -rn --include='*.tsx' --include='*.ts' -oE '(title|placeholder|aria-label|alt)="[^"]+"' <dir>
   ```
   The grep is a starting point — read each file to confirm what surfaces to the user.

2. **Draft the namespace JSON.** Add every extracted string to `frontend/src/locales/en/<namespace>.json` and `frontend/src/locales/fr/<namespace>.json` in mirror shape. Key naming is semantic — use nested paths that reflect the UI hierarchy (`header.title`, `sections.moyennes.title`, `actions.save`). NEVER use the English or French text as the key.

3. **Wire `useTranslation`.** In each modified file:
   ```tsx
   import { useTranslation } from 'react-i18next';
   // …
   const { t } = useTranslation('<namespace>');
   ```
   Replace the FR literal with `{t('<key>')}` in JSX or `t('<key>')` in expressions.

4. **Handle interpolation & pluralization.**
   - Interpolation: `t('imports.summary', { count })` → `"summary": "{{count}} transactions imported"`.
   - Pluralization (i18next built-in): `"summary_one": "{{count}} transaction imported"`, `"summary_other": "{{count}} transactions imported"`. FR side gets `summary_one`, `summary_other` with matching French plural forms.

5. **Update tests.** For assertions that rely on French literals:
   - **Preferred**: rewrite as locale-neutral (regex on digits, `data-testid`, structural asserts).
   - **Fallback**: add a second render pass with `i18n.changeLanguage('en')` and assert the English equivalent.
   - Do NOT delete existing FR-literal tests wholesale — they still catch regressions in the default (`fr`) locale.

6. **Verify.**
   ```bash
   cd frontend
   npm test -- --run <affected-test-glob>   # focused
   npm test -- --run                          # full suite before commit
   npm run build                              # tsc + vite build
   ```
   Then manually: `npm run dev`, open the migrated page in the browser, switch to English via the globe icon in the header, confirm no French leakage; switch back to French, confirm no regressions.

7. **Commit** with the message `i18n(<namespace>): extract <feature> strings` and the Gekkotron identity flags.

- [ ] **Step 2: Create `frontend/src/i18n/index.ts`**

```ts
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import LanguageDetector from 'i18next-browser-languagedetector';

const NAMESPACES = [
  'common',
  'layout',
  'dashboard',
  'transactions',
  'imports',
  'rules',
  'accounts',
  'budgets',
  'settings',
  'pdf-template',
  'charts',
  'tips',
] as const;

type Ns = (typeof NAMESPACES)[number];

async function loadNamespace(lang: 'en' | 'fr', ns: Ns) {
  return (await import(`../locales/${lang}/${ns}.json`)).default;
}

i18n
  .use(LanguageDetector)
  .use(initReactI18next)
  .init({
    fallbackLng: 'en',
    supportedLngs: ['en', 'fr'],
    ns: ['common'],
    defaultNS: 'common',
    interpolation: { escapeValue: false },
    detection: {
      order: ['localStorage', 'navigator'],
      lookupLocalStorage: 'athena.lang',
      caches: ['localStorage'],
    },
    resources: {},
    partialBundledLanguages: true,
  })
  .then(async () => {
    // Load the currently active language's common bundle immediately so the
    // first paint has strings; other namespaces lazy-load per useTranslation.
    const current = (i18n.language.startsWith('fr') ? 'fr' : 'en') as 'en' | 'fr';
    const common = await loadNamespace(current, 'common');
    i18n.addResourceBundle(current, 'common', common, true, true);
  });

// Register a backend that fetches namespace bundles on demand.
i18n.services.backendConnector.backend = {
  type: 'backend',
  init: () => {},
  read: async (lng: string, ns: string, callback: (err: unknown, data?: unknown) => void) => {
    try {
      const lang = (lng.startsWith('fr') ? 'fr' : 'en') as 'en' | 'fr';
      const bundle = await loadNamespace(lang, ns as Ns);
      callback(null, bundle);
    } catch (err) {
      callback(err);
    }
  },
} as unknown as typeof i18n.services.backendConnector.backend;

export default i18n;
export type { Ns };
```

Note: if the backendConnector pattern above turns out incompatible with the installed i18next major, fall back to eagerly-importing every namespace via a static `resources` map in `.init(…)`. The eager form is fine for a ~12-namespace app (bundle size impact ≈ 20–40 KB gzipped). If you switch to eager mode, note it in the report — do NOT introduce a third loading strategy.

- [ ] **Step 5: Create `frontend/src/i18n/LanguageSwitcher.tsx`**

```tsx
import { useTranslation } from 'react-i18next';

const LANGS = [
  { code: 'en', label: 'English' },
  { code: 'fr', label: 'Français' },
] as const;

export function LanguageSwitcher() {
  const { i18n } = useTranslation();
  const current = (i18n.language?.startsWith('fr') ? 'fr' : 'en') as 'en' | 'fr';

  return (
    <label className="inline-flex items-center gap-1 text-sm">
      <span aria-hidden="true">🌐</span>
      <select
        value={current}
        onChange={(e) => {
          const next = e.target.value as 'en' | 'fr';
          void i18n.changeLanguage(next);
          try {
            localStorage.setItem('athena.lang', next);
          } catch {
            // storage may be blocked in private mode; the change still applies
            // to this session.
          }
        }}
        className="bg-transparent focus:outline-none"
        aria-label="Language"
      >
        {LANGS.map((l) => (
          <option key={l.code} value={l.code}>
            {l.label}
          </option>
        ))}
      </select>
    </label>
  );
}
```

Emoji fallback for the globe icon is intentional — no external icon dependency, works in every browser. If Athena's design system already ships a globe icon (check `NavIcons.tsx`), swap it in; otherwise stay with the emoji.
